# NPC 救援 + 元进度设计

> 状态：v1（2026-07-24）。基调：地图中解救被困幸存者 → NPC 加入据点提供服务 → 跨局永久解锁。
> 关联：救援点插槽见 docs/mapgen-design.md；服务经济见 docs/economy-design.md；叙事框架见 docs/game-design.md。
> 决策前提（用户定）：**完整纳入**（非最小切片）、**跨局元进度解锁**、**就地解救**（不做护送）。带 ❓ 为试玩/内容期待定。

## 一句话

战斗图里偶尔困着一个**幸存者 NPC**，被尸群围住等死。清掉围困的尸群即解救——TA 会前往安全据点（休整/商店图），此后为你开一类服务；救过的 NPC **跨局永久解锁**，让后续每一局的据点服务更全。这把"探索动机、经济消费出口、叙事温度、局外成长"拧成一股。

## 三个设计支柱对齐

- **压柱 1（取舍）**：一张战斗图最多困 1 个 NPC，救 TA 要清一片尸群、耗时间耗血——"救不救、值不值"是又一个搜打撤微决策。救援点和资源点/载具争夺同一份图内停留预算。
- **压柱 2（操作简单）**：解救 = 把围困尸群清空，无额外按键、无护送操作。NPC 得救后自动离场（叙事上"搭你之前的载具走"），不占用玩家操作。
- **叙事温度**：接应点承诺救援 → 你反过来在废墟里救人 → 据点从空屋变成有人烟的营地。元进度不是冷冰冰的数值解锁，是"你救回来的人"。

## 救援机制（局内）

### 救援点
- **来源**：mapgen 战斗图模块新增 `rescue` 插槽（每模块 0~1 个，稀有）；每张战斗图 **0~1 个救援点**（`spawn_chance` ❓ 40%），拼装时从可用 rescue 插槽随机启用。精英图必出 1 个（高风险高回报主题一致）。
- **呈现**：一个**被困 NPC**（cuddlebug idle 帧染成区别色）站在原地不动，周围 `GUARD_RADIUS`(≈180px) 内预置一圈**守卫尸群**（6~10 只，比常规略密）。迷雾下靠近才发现；发现后迷你图标记（复用 discovered 机制，青色点）。
- **NPC 头顶显示身份图标 + "解救：清除周围尸群"提示。**

### 解救条件
- 玩家进入 NPC 的 `GUARD_RADIUS` 触发"交战"状态：只要该半径内的**活尸数量降到 0**，即判定解救成功。
- 守卫尸群是**预置的固定编队**（不随 HeatDirector 持续刷新计入——否则永远清不完）；但 HeatDirector 的常规刷怪仍会游荡过来，可能混入，需玩家把区域彻底清空。❓ 若持续刷怪导致难以清零，改为"预置守卫全灭即解救"（只认编队 tag，不认游荡尸）——**MVP 采后者，稳**。
- 解救瞬间：NPC 播一个上车/挥手动画，`rescued` 信号，淡出离场；掉一份即时奖励（30~50 金 + 1 蓝档强化拾取物，肯定 NPC 感激）；记入 `RunState.rescued_this_run`。

### 与压力/经济的咬合
- 守卫尸群密度 = 一次小型 loot_pressure 级交战，天然是"贪不贪"的决策：清完约耗 20~40 秒 + 若干血，换 NPC 解锁 + 即时奖励。
- 救援点与资源点共享"图内停留预算"——多花时间救人 = 少搜一个箱子 or 更晚找到载具（压柱 1）。

## NPC 类型与服务

每个 NPC 绑一类**据点服务**。得救的 NPC 出现在**安全图**（休整/商店）提供服务；未解锁的 NPC 对应服务不出现（或以"空缺工位"占位，提示"去救 XX"）。

| NPC | 服务 | 出现图 | 说明 |
|---|---|---|---|
| **老兵（商店老板）** | 物资自动兑换 + 货架（绷带/扩容） | 商店 | 现有商店服务归 TA；未解锁则商店只兑换不开架❓ |
| **医师** | 治疗（20 金/30 血），解锁后**治疗效率↑**或解锁"急救包"货架 | 休整 | 现有治疗归 TA |
| **军械师** | 武器箱/副武器货架、武器精通重随 | 商店/休整 | 让"搜武器"多一个稳定来源 |
| **线人（无线电员）** | 出图前**预览下一层候选图的奖励倾向**（game-design 待定的"第二信息维度"） | 休整 | 信息型服务，呼应压柱 1 |
| **机械师** | 载具相关：降低载具收尾时间 / 解锁"额外候选图"❓ | 休整 | 内容期扩展 |

- **MVP 切片**：先做 **3 个 NPC**——老兵（商店）、医师（治疗）、军械师（武器货架）。线人/机械师内容期补。
- 现有休整/商店服务**不删**：未解锁对应 NPC 时保持现状（服务照常，视作"你自己凑合"）；解锁后由 NPC "接管"并**增强**（更好定价/额外货架/额外选项）——保证"没救到人也能玩，救到人更爽"，不惩罚新玩家。

## 跨局元进度（存盘）

- **存盘文件**：`user://meta.cfg`（ConfigFile），独立于局内 RunState（局内死亡全丢的紧张感不受影响）。
- **存什么**：`rescued_npcs`（已永久解锁的 NPC id 集合）、`total_runs`/`total_extractions`（统计，成就用）。**只存"救过谁"，不存货币/物资**——撤离带出的货币仍是局内结算展示，不进元进度池（避免"搬砖"腐蚀 roguelike 张力）。
- **解锁规则**：一局内成功解救某 NPC → 撤离成功后写入 meta 永久解锁 ❓（或"救到即解锁，不必撤离成功"——**MVP 采"救到即解锁"**，更宽容，鼓励救人；死亡也保留已救的人，唯一代价是本局货币丢失）。
- **解锁效果**：下一局起，该 NPC 的服务在安全图**默认可用/增强**。已解锁的 NPC **仍会在战斗图出现**（可重复救，给即时奖励，但不重复解锁——重复救的价值是即时金币+强化）。
- **AutoLoad**：新增 `MetaProgress` autoload（读写 meta.cfg，暴露 `is_unlocked(npc_id)` / `unlock(npc_id)` / 统计）。RunState 只管局内，MetaProgress 管跨局——职责分离（tech-design §2 精神）。

## 与"死亡全丢"的边界（关键）

- 局内货币/物资/等级：死亡全丢，**不变**。
- NPC 解锁：跨局保留，**是唯一的局外成长**。它不给数值加成（不破坏局内平衡曲线），只**扩充服务可用性**——即"内容解锁"而非"数值膨胀"。这是把元进度做成"横向内容"而非"纵向碾压"，避免老手碾压、新手劝退。
- ❓ 是否给微弱数值元进度（如"救满 5 人解锁新起始角色"）——呼应 game-design 待定的"解锁角色/永久强化"，**内容期定，MVP 只做服务解锁**。

## MVP 切片（NPC 部分）

- mapgen：战斗图 `rescue` 插槽 + 救援点生成（每图 0~1，精英图必出）
- RescuePoint 实体：被困 NPC + 预置守卫编队 + 清零判定 + 解救奖励/离场
- 3 个 NPC id（veteran/medic/armorer）+ MetaProgress autoload（user://meta.cfg）
- 安全图服务按解锁状态增强（老兵接管商店货架、医师接管治疗、军械师加武器货架）
- 主菜单/结算显示"已救援 N 位幸存者"（元进度可见）
- 验收问题：清尸群救人的过程值得吗？"据点里多一个人"有满足感吗？未解锁时的"空工位"是否让人想去救？
